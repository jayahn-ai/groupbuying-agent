작성일: 2026년 4월 25일

## 에이전트 도입 배경 및 측정 목적

공동구매 업무는 인플루언서 섭외 → 캠페인 세팅 → 공구 진행 → 정산 계산 → 메일 발송까지 수 일에 걸쳐 선형적으로 진행된다. 각 단계마다 담당자가 Google Sheets를 열고, 계산하고, 메일을 발송하는 반복적 수작업이 누적된다. 브랜드 매출에 직접적으로 기여하는 퍼포먼스 마케팅 업무임에도 불구하고 단순 반복 업무의 비중이 높아 퍼포먼스 마케터가 전략적 판단에 집중하지 못하는 구조적 문제가 있다.

이 에이전트는 해당 선형 업무 전체를 웹훅 기반 챗봇 인터페이스로 자동화한다. KPI는 크게 세 가지 질문에 답하기 위해 설정한다:

1. 에이전트의 답변과 계산 결과를 신뢰할 수 있는가? (정확도·품질)
2. 시스템이 안정적으로 운영되고 있는가? (신뢰성)
3. 에이전트가 사람의 일을 실제로 줄이고 있는가? (자동화 효율)

## KPI 1 — 인텐트 분류 정확도

정의: 웹훅으로 들어온 사용자 메시지가 올바른 분기(engagement / simulation / settlement / settlement_confirm / rag)로 라우팅된 비율

측정 방법:
```
정확도 = 올바르게 분류된 요청 수 / 전체 요청 수 × 100
```
n8n 실행 API(`/api/v1/executions`)에서 `lastNodeExecuted` 필드를 확인하여 의도한 분기로 진행됐는지 샘플링 검토.

목표치: ≥ 95%

측정 주기: 주간

선정 이유:  
인텐트 분류는 에이전트 전체 흐름의 입구다. 이 단계에서 오분류가 발생하면 정산 요청이 RAG로 가거나, 단순 질의가 메일 발송 플로우로 흘러 실제 Gmail이 발송되는 심각한 오작동이 발생한다. 특히 "정산 결과 보여줘(settlement)"와 "정산 메일 보내줘(settlement_confirm)"는 문장이 유사하지만 결과가 전혀 다르므로 분류 정확도가 비즈니스 안전성과 직결된다. 95% 목표는 주 100건 기준 주 5건 이하 오분류를 허용하는 수준이며, 초과 시 Gemini 프롬프트 개선 혹은 워크플로우 수정으로 유지보수 작업을 진행한다.

## KPI 2 — RAG 할루시네이션 발생률

정의: RAG 플로우에서 지식 베이스에 없는 내용을 에이전트가 지어내거나, 정답이 없는 질문에 모른다고 답하지 않는 비율

측정 방법:
```
헛답변율 = 헛답변 건수 / RAG 요청 전체 건수 × 100

측정 방법:
1. 월간 사용자 쿼리 기준 
2. 에이전트가 "모르겠습니다" 또는 "정보가 없습니다"로 답하는지 확인
3. 월 1회 자동 측정 + 이상값 수동 검토
```

목표치: ≤ 5%

측정 주기: 월간

선정 이유:  
공동구매 에이전트는 계약 조건, RS율 정책, 최소 판매 수량 등 정확한 정보가 중요한 영역을 다룬다. 에이전트가 확인되지 않은 정보를 자신 있게 말하면 인플루언서가 잘못된 판단을 내릴 수 있다. 시스템 프롬프트에 "검색 결과가 없으면 모른다고 답하라"는 규칙이 있지만, 이를 실제로 준수하는지 정기 검증이 필요하다. 헛답변율이 5%를 초과하면 벡터 DB 커버리지를 점검하고 시스템 프롬프트를 강화한다.

## KPI 3 — 웹훅 요청 성공률

정의: 전체 웹훅 요청 중 n8n 실행 상태가 `success`인 비율

측정 방법:
```
성공률 = success 실행 수 / 전체 실행 수 × 100

측정 쿼리:
GET /api/v1/executions?workflowId=iEr5iv4VpURCS303&limit=100
→ status별 집계
```

목표치: ≥ 98%

측정 주기: 일간 (n8n 실행 API 자동 집계)

선정 이유:  
에이전트는 퍼포먼스 마케터가 업무 중 실시간으로 사용하는 도구다. 요청 실패가 잦으면 신뢰가 무너지고 수동 업무로 회귀하게 된다. 성공률 98%는 일 10건 기준 월 5건 이하 실패를 허용하는 수준이다. 실패 원인은 크게 세 가지(외부 API 오류, 코드 노드 에러, 데이터 파싱 실패)로 분류되며, 반복 실패 패턴이 감지되면 해당 노드에 에러 핸들링을 추가한다.

## KPI 4 — 평균 응답 시간

정의: 웹훅 요청 접수(`startedAt`)부터 respondToWebhook 완료(`stoppedAt`)까지의 평균 소요 시간

측정 방법:
```
응답 시간 = stoppedAt - startedAt (초 단위)
평균 = Σ 응답 시간 / 측정 건수

n8n 실행 API에서 startedAt, stoppedAt 필드 수집
```

목표치: ≤ 15초 (평균), ≤ 30초 (95% 이내)

측정 주기: 월간

선정 이유:  
공동구매 에이전트의 각 플로우는 Google Sheets 읽기, Gemini API 호출, PostgreSQL 조회 등 외부 API를 다수 포함하나, 실시간 대기 중 응답이 늦으면 에이전트를 사용하지 않게 된다. 플로우별 평균을 비교하여 RAG 플로우(AI Agent 포함)가 가장 느린 경우 프롬프트 최적화나 캐싱 전략을 적용한다.

플로우별 예상 응답 시간:

| 플로우 | 예상 시간 |
|---|---|
| 인게이지먼트 분석 | 5~10초 |
| 수익 시뮬레이션 | 5~10초 |
| 정산 계산 | 10~20초 |
| 정산 메일 발송 | 15~25초 |
| RAG 질의응답 | 5~10초 |

## KPI 5 — 정산 메일 발송 누락률

**정의:** 정산 사이클 종료 후 `정산 메일 발송 여부` 컬럼이 "미발송"으로 남아있는 비율

**측정 방법:**
```
누락률 = 미발송 상태 행 수 / 전체 정산 행 수 × 100

측정: 정산 사이클 종료 D+2일에 Google Sheets 공구 정산 기록 시트 확인
```

**목표치:** = 0% (단 1건의 누락도 허용하지 않음)

**측정 주기:** 정산 사이클마다 (월 1~2회)

**선정 이유:**  
정산 메일 누락은 인플루언서와의 신뢰 훼손으로 직결된다. 인플루언서 입장에서 정산 지연은 수익 미지급과 동일하게 느껴지며, 반복 시 이탈로 이어진다. 이 지표는 "에이전트가 실수 없이 끝까지 처리했는가"를 측정하는 업무 완결성 KPI다. 목표치를 0%로 설정한 이유는 정산 금액이 계약 의무 사항이기 때문이다. 누락 감지 시 당일 수동 발송 후 에이전트 코드를 점검한다.

---

## KPI 6 — 정산 처리 사이클 타임 단축률

정의: 에이전트 도입 전(수동)과 도입 후(에이전트) 정산 완료까지 소요된 시간 비교

측정 방법:
```
수동 처리 시간: 주문 집계 → 계산 → 시트 입력 → 메일 작성 → 발송 (담당자 직접 측정)
에이전트 처리 시간: 웹훅 요청 → 정산 기록 저장 → 메일 발송 완료

단축률 = (수동 시간 - 에이전트 시간) / 수동 시간 × 100
```

목표치: ≥ 80% 단축 (수동 2시간 → 에이전트 24분 이내)

측정 주기: 월간 (첫 3개월간 수동 측정 병행)

선정 이유:  
이 에이전트의 존재 이유 자체가 "반복적 수작업을 대신한다"는 것이다. 정산 업무는 주문 데이터 집계, RS율 계산, 시트 입력, 메일 작성까지 숙련된 담당자도 15분 이상 소요된다. 에이전트는 동일 작업을 웹훅 1회 호출로 처리한다. 이 지표는 에이전트의 ROI를 정량화하는 가장 직접적인 근거가 되며, 도입 결정자에게 "에이전트를 계속 운영해야 하는 이유"를 수치로 제시할 수 있다.
